
## **Práctica 1 — Configuración del proyecto y primera API**

**Objetivo:** Montar un proyecto Django con Django REST Framework y listar recursos básicos.

### Actividades

1. Crear proyecto Django.
2. Crear app `quickstart`.
3. Instalar y configurar DRF (`INSTALLED_APPS`).
4. Ejecutar migraciones y crear superusuario.
5. Definir serializadores para `User` y `Group`.
6. Crear viewsets para `User` y `Group`.
7. Registrar rutas con **routers**.
8. Probar la API en el navegador y con CLI (`curl` o `httpie`).

### Resultado esperado

* Endpoints `/users/` y `/groups/` funcionales.
* Navegable desde el **Browsable API** de DRF y protegido con autenticación básica. ([django-rest-framework.org][1])

## **Práctica 2 — Serialización y validación de datos**

**Objetivo:** Entender cómo los *serializers* transforman los modelos en JSON.

### Actividades

1. Crear un modelo sencillo (por ejemplo, `Producto` con campos `nombre`, `precio`, `disponible`).
2. Definir un `ModelSerializer` para el modelo.
3. Añadir reglas de validación:

   * Nombre obligatorio, mínimo 3 caracteres.
   * Precio mayor que 0.
4. Probar la creación de recursos con `POST` y validar errores.

### Resultado esperado

* Serializer que valide datos eficientemente antes de guardar.
* Errores bien estructurados retornados en JSON.

> Aquí se aprende qué hace un **serializer** y cómo validar datos en una API. ([django-rest-framework.org][2])
> Aquí cómo funcionan y se usan las validaciones de campos. ([wiki-Serializers][7])

## **Práctica 3 — Requests y Responses**

**Objetivo:** Manejar diferentes tipos de solicitudes desde la API.

### Actividades

1. Crear un endpoint con `@api_view` para métodos `GET` y `POST`.
2. Usar `request.data` para leer datos enviados desde el cliente.
3. Devolver respuestas con distintos códigos HTTP (`200`, `201`, `400`).
4. Probar con diferentes clientes (POSTMAN / HTTPie / curl).

### Resultado esperado

* Endpoint que reciba JSON y responda con JSON correctamente.
* Entender cómo DRF reemplaza a `HttpRequest` y `JsonResponse`. ([django-rest-framework.org][3])

## **Práctica 4 — Autenticación y permisos**

**Objetivo:** Agregar control de acceso a la API.

### Actividades

1. Configurar autenticación básica y por sesión en `settings.py`.
2. Probar que las rutas requieren login para modificar datos.
3. Crear permisos personalizados: solo usuarios autenticados pueden crear/editar.
4. Bloquear accesos no autorizados.

### Resultado esperado

* Usuarios anónimos pueden leer pero no modificar.
* Solo usuarios autenticados pueden `POST`, `PUT`, `DELETE`. ([django-rest-framework.org][4])

## **Práctica 5 — Relaciones y APIs hiperconectadas**

**Objetivo:** Entender cómo representar relaciones en la API y usar URLs hiperlinkadas.

### Actividades

1. Añadir un modelo relacionado (por ejemplo, `Comentario` a un `Producto`).
2. Usar campos de relación (`HyperlinkedRelatedField`) en los serializers.
3. Crear endpoint raíz que refleje hiperenlaces (links a todos los recursos).
4. Añadir vista para una representación HTML (si procede).

### Resultado esperado

* API con relaciones bien estructuradas.
* Camino hacia API más RESTful e hipertextual. ([django-rest-framework.org][5])

## **Práctica 6 — ViewSets y Routers**

**Objetivo:** Refactorizar *views* básicas a *ViewSets* y usar *routers* automáticos.

### Actividades

1. Refactorizar tus vistas `GET`, `POST`, etc. a un `ModelViewSet`.
2. Registrar los ViewSets con `DefaultRouter`.
3. Eliminar rutas declaradas manualmente para reemplazar por las automáticas.
4. Comparar el resultado con la versión anterior.

### Resultado esperado

* Endpoints CRUD creados automáticamente por DRF.
* URLs bien organizadas con poco código. ([django-rest-framework.org][6])

## **Práctica 7 — Paginación y configuración avanzada**

**Objetivo:** Mejorar la API para que devuelva resultados paginados.

### Actividades

1. Añadir en `settings.py`:

   ```python
   REST_FRAMEWORK = {
     "DEFAULT_PAGINATION_CLASS": "rest_framework.pagination.PageNumberPagination",
     "PAGE_SIZE": 10,
   }
   ```
2. Probar que los endpoints listan resultados paginados.
3. Personalizar `PAGE_SIZE` y probar distintos valores.

### Resultado esperado

* Resultado paginado con `next`, `previous`, `count` y `results`.

## **Práctica 8 — Pruebas básicas de API**

**Objetivo:** Crear tests que verifiquen el correcto funcionamiento de la API.

### Actividades

1. Crear tests con **APITestCase**.
2. Testear:

   * Listado de recursos.
   * Creación con credenciales correctas.
   * Rechazo de creación sin autenticación.
3. Ejecutar tests con `pytest` o `manage.py test`.

### Resultado esperado

* Conjunto de tests que certifican el funcionamiento de los endpoints.
